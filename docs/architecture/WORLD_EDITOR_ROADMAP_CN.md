# 世界编辑器路线图

## 目标

把当前的 sandbox/editor shell 从“偏物理演示工具”推进成一个具备稳定创作闭环的世界编辑器。

当前代码库已经具备这些基础：

- 编辑器外壳 UI
- 层级 / 属性 / 调试面板
- PIE 生命周期
- 场景切换与场景级元数据编辑
- 命令与历史记录基础设施
- project/workspace/prefab/asset pipeline 基础能力

下一阶段的重点不应该继续堆零散快捷键或一次性的 sandbox 行为，而应该把这些基础真正收束成可持续的编辑工作流。

## 当前所处阶段

这个仓库已经不只是一个物理 sandbox。

它已经具备“早期世界编辑器”特征：

- `apps/sandbox_dwrite/` 下已经有编辑器外壳
- runtime / editor 分层已经开始建立
- PIE 进入 / 暂停 / 单步 / 退出闭环已经存在
- 场景重命名 / 排序 / 资产 GUID 编辑已经存在
- undo / redo 基础设施已经存在
- project workspace、package/plugin、prefab semantics、asset pipeline 这些系统已经存在

但距离一个真正可用的世界编辑器，还缺少以下核心能力：

- 持久化的场景编辑闭环
- 完整命令化的编辑路径
- 视口创作工具
- prefab 实例工作流
- 资源浏览与绑定工作流

## 优先级

### P0：必须先做

这些事情应该优先于继续扩展编辑器表面功能。

1. 统一编辑数据模型

- 明确定义编辑器侧的数据概念：
  - `World`
  - `Scene`
  - `Entity` / `Body`
  - `Constraint`
  - `AssetRef`
  - `PrefabInstance`
- 减少 UI 状态与 runtime 对象直接耦合。
- 让编辑态成为唯一真源，runtime 作为执行副本。

2. 完成场景持久化工作流

- 增加：
  - 新建场景
  - 保存
  - 另存为
  - dirty 标记
  - 切换前修改确认
- 把现在的 preset scene 方式推进为真正可编辑的 scene 文件工作流。

3. 所有编辑操作统一走 command/history

- 保证所有用户可见编辑都走 command bus + history transaction：
  - 创建
  - 删除
  - 重命名
  - 排序
  - 参数修改
  - 资源绑定
- 避免存在绕过 undo/redo 的直接修改路径。

4. 保持文档与行为一致

- 帮助文案、quickstart、实际快捷键必须同步。
- 编辑器交互说明本身就是功能的一部分，不能永远靠“之后再补”。

### P1：高价值编辑器能力

这些能力会把当前外壳真正变成可用创作工具。

1. 视口编辑工具

- 平移 / 旋转 gizmo
- 吸附
- 框选
- 多选
- duplicate / copy / paste

2. 层级面板升级

- 搜索 / 过滤优化
- 拖拽排序
- 右键菜单
- 批量操作
- 类型 / 分组过滤

3. 属性面板升级

- 组件化分组展示
- 更稳定的数值输入体验
- 重置默认值
- 多选时 mixed-value 显示

4. Prefab 工作流

- 实例化 prefab
- 显示 override 状态
- apply overrides
- revert overrides
- 跳转 prefab 源

5. 资源浏览器

- 不再依赖手填 GUID
- 从 UI 中选择并绑定资源
- 显示基础元信息与依赖提示

### P2：后续扩展

这些方向重要，但不应阻塞核心编辑闭环。

1. 大世界支持

- chunk 化
- streaming
- 场景分区
- 分层可见性

2. 脚本化工具

- 批处理编辑工具
- 迁移工具
- 自动修复工具

3. 校验与诊断

- scene validation 面板
- 编辑期错误报告
- 更强的恢复机制

4. 发布工作流

- 独立 runtime/player
- 项目模板
- 打包配置

## 推荐执行顺序

1. 场景保存 / 加载 / dirty 工作流
2. 全量命令化编辑路径
3. 视口 gizmo 与多选
4. prefab 实例工作流
5. 资源浏览器与绑定体验

## 为什么这样排序

当前仓库最缺的并不是更多面板，而是一个可靠的编辑闭环：

- 产生修改
- 持久化
- undo / redo
- 安全进入 PIE
- 退出 PIE 后继续编辑

如果这条链路不稳定，继续扩展更多编辑器表面功能，只会让复杂度增长得比能力更快。

## 进展判断标准

当以下条件都成立时，可以认为编辑器真正进入了更成熟的“世界编辑器阶段”：

- scene 是真实可编辑资产，而不只是 preset 切换
- 主要编辑操作都能通过统一命令路径撤销 / 重做
- 视口编辑不再依赖调试式或控制台式流程
- prefab 实例有可见、稳定的创作闭环
- 资源绑定通过编辑器 UI 完成，而不是靠手填 GUID

## 近期最优建议

最值得立刻推进的下一步是：

1. 显式定义 editor-side 的 scene/world 模型
2. 完成 scene save/load/dirty 处理
3. 清理剩余绕过 command/history 的直接修改路径

这一顺序会为后续所有世界编辑器功能打下真正稳定的基础。
